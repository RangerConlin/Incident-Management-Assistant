Source of truth inspected:
C:\Users\Brendan\Documents\GitHub\Incident-Management-Assistant\cloud_server

Server runtime:
- FastAPI application is created in cloud_server\sarapp_db\api\app.py
- Cloud server listens on port 8765 by default
- Mobile MVP hardcoded cloud base URL: http://srv1707346.hstgr.cloud:8765
- Health check: GET /health
- Server metadata: GET /server-info
- Mongo URI is configured with SARAPP_MONGO_URI

Current MongoDB structure:
- sarapp_system: server/app state and incident registry
- sarapp_master: agency-wide reference data such as personnel, certifications, organizations, vehicles, aircraft, equipment, resource types, hazard types, radio channels, templates, and user profile scaffolding
- sarapp_incident_<incident_id>: per-incident operational data such as incident profile, teams, tasks, check-ins, check-in history, resources, comms logs, ICS forms, GIS, planning, logistics, safety, intel, approvals, and audit logs

Important terminology update:
- Use "incident", not "mission", for API contracts.
- Current server does not expose POST /api/mobile/login or JWT/token auth.
- Current mobile MVP should treat a successful GET /health as server connection.
- If mobile auth is added later, it should be designed as a new server feature and documented separately.

Mobile-owned responder profile:
- Stored locally on the device using AsyncStorage.
- Includes personal info, certifications, assigned vehicles, and assigned equipment.
- During incident check-in, the profile is copied into the check-in request as mobile_profile_snapshot.
- The same request also sends common roster fields such as display_name, incident_phone, incident_callsign, role_on_team, team_id, vehicles, and equipment.
- Current target collection: sarapp_incident_<incident_id>.checkins.
